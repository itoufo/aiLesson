## 3.2 WBS（Work Breakdown Structure）

### 3.2.1 WBSとは

**WBS**は、プロジェクトの成果物を階層的に分解した構造図です。

![WBS構造](/docs/images/02_応用/03_wbs_structure.jpeg)

### 3.2.2 WBS作成プロンプト

```markdown
以下のプロジェクトのWBSを作成してください。

## プロジェクト概要
- プロジェクト名：[名称]
- 目的：[何を達成するか]
- 成果物：[最終的に何ができあがるか]
- 期間：[概算期間]
- 制約条件：[予算、リソース、技術的制約など]

## WBS作成の条件
- レベル3まで分解
- 各タスクは具体的かつ測定可能に
- 100%ルール（全タスクの合計 = プロジェクト全体）を適用

## 出力形式

### WBS（階層形式）

1. [フェーズ1名称]
   1.1 [タスク名]
       - 成果物：[成果物]
       - 概算工数：[人日]
   1.2 [タスク名]
       ...

2. [フェーズ2名称]
   2.1 [タスク名]
   ...

### WBS辞書（主要タスクの詳細）

| WBS番号 | タスク名 | 成果物 | 担当 | 工数 | 依存関係 |
|---------|----------|--------|------|------|----------|
| 1.1 | | | | | |
| 1.2 | | | | | |
```

### 3.2.3 WBS作成の実例

```markdown
【プロンプト例】
社内ポータルサイトリニューアルプロジェクトのWBSを作成してください。

プロジェクト概要：
- 目的：10年前に構築した社内ポータルを刷新
- 期間：6ヶ月
- 主な機能：お知らせ掲示、社員検索、申請ワークフロー、FAQ
- 制約：既存データの移行必須、AWS上で構築

---

【AI出力例】

### WBS

1. プロジェクト管理
   1.1 プロジェクト計画策定
   1.2 進捗管理・報告
   1.3 品質管理

2. 要件定義
   2.1 現状分析
       2.1.1 既存システム調査
       2.1.2 ユーザーヒアリング
   2.2 要件整理
       2.2.1 機能要件定義
       2.2.2 非機能要件定義
   2.3 要件レビュー・承認

3. 設計
   3.1 基本設計
       3.1.1 システム構成設計
       3.1.2 画面設計
       3.1.3 データベース設計
   3.2 詳細設計
       3.2.1 機能別詳細設計
       3.2.2 インターフェース設計

4. 開発・構築
   4.1 インフラ構築
   4.2 機能開発
       4.2.1 お知らせ機能
       4.2.2 社員検索機能
       4.2.3 ワークフロー機能
       4.2.4 FAQ機能
   4.3 結合テスト

5. データ移行
   5.1 移行計画策定
   5.2 移行ツール開発
   5.3 移行リハーサル
   5.4 本番移行

6. テスト・リリース
   6.1 総合テスト
   6.2 ユーザー受入テスト
   6.3 リリース準備
   6.4 本番リリース
   6.5 安定化運用
```

## 3.4 KPI設定と進捗管理

### 3.4.1 KPIとは

**KPI**（Key Performance Indicator：重要業績評価指標）は、目標達成度を測定する指標です。

![KPI設定のSMART原則](/docs/images/02_応用/03_smart_kpi.jpeg)

### 3.4.2 KPI設定プロンプト

```markdown
以下のプロジェクトのKPIを設定してください。

## プロジェクト概要
- プロジェクト名：[名称]
- 目的：[目的]
- 期間：[期間]
- 主要な成果物：[成果物]

## KPI設定の条件
- SMART原則に従う
- プロジェクトKPIと運用KPIを分けて設定
- 先行指標（リード）と結果指標（ラグ）を含める

## 出力形式

### プロジェクトKPI（プロジェクト期間中の指標）

| KPI名 | 定義 | 目標値 | 測定方法 | 測定頻度 | 種別 |
|-------|------|--------|----------|----------|------|
| | | | | | 先行/結果 |

### 運用KPI（リリース後の効果測定指標）

| KPI名 | 定義 | 目標値 | 測定方法 | 測定頻度 |
|-------|------|--------|----------|----------|
| | | | | |

### KPIツリー
[上位KPIと下位KPIの関係を図示]
```

### 3.4.3 進捗報告プロンプト

```markdown
以下の情報を基に、プロジェクト進捗報告書を作成してください。

## 進捗データ

## 2. マイルストーン状況

| マイルストーン | 予定日 | 実績/見込 | 状態 |
|----------------|--------|-----------|------|
| | | | ◎/○/△/× |

|-----|------|------|--------|------|

|------|--------|------|------|------|

## 5. 次期の計画
[次の報告期間の計画]
```

## 3.5 リスク管理

### 3.5.1 リスク分析プロンプト

```markdown
以下のプロジェクトのリスク分析を行ってください。

## プロジェクト概要
[プロジェクトの概要]

## リスク分析の観点
- 技術リスク
- スケジュールリスク
- リソースリスク
- 外部リスク
- ビジネスリスク

## 出力形式

### リスク一覧

| No | リスク内容 | カテゴリ | 発生確率 | 影響度 | リスクスコア | 対応策 |
|----|------------|----------|----------|--------|--------------|--------|
| 1 | | | 高/中/低 | 高/中/低 | (確率×影響) | |

### リスクマトリクス
```

   └──────────────┴──────────────┴──────────────
          低            中            高
                    発生確率

## 章末まとめ

### 本章のポイント

1. **WBS**でプロジェクトを階層的に分解し、漏れなく計画
2. **ガントチャート**でスケジュールを可視化
3. **KPI**はSMART原則で具体的・測定可能に設定
4. **リスク管理**で発生確率×影響度でリスクを評価
5. AIを活用して計画・報告書作成を効率化
